🛢️ Contexto del negocio
¿Por qué pronosticar producción de pozos?
⌄
Decisiones de inversión, mantenimiento y compromisos comerciales dependen de saber cuánto va a producir cada pozo en los próximos meses.

  • Trazabilidad de modelos: auditabilidad, linaje, reproducibilidad.

📊 Estructura del dataset crudo
CSV de pozos: qué columnas, qué problemas
⌄
Un registro por pozo por mes. Variables operativas + producciones de gas/petróleo/agua.
🗂️ Columnas usadas
Identificación: idpozo
Fecha: anio + mes → se compone a fecha tipo datetime
Targets (numéricos): prod_gas, prod_pet, prod_agua
Variables estáticas: tef (tiempo efectivo), profundidad
Categóricas: tipoextraccion, tipopozo, provincia, cuenca
🛠️ Preprocesamiento aplicado
  • anio + mes → fecha (datetime, primer día del mes).
  • dropna(subset=["idpozo","fecha"]) — sin esos campos no hay registro útil.
  • Categóricas → LabelEncoder a int32.
  • Targets + numéricas → float32.
  • Sort por (idpozo, fecha) para poder calcular lags por grupo.
FALSO"Las features están normalizadas a media 0 y varianza 1." → No, los Random Forest no necesitan scaling. Solo se hace casting de tipo.
FALSO"Cada pozo tiene un solo registro." → No, hay un registro por pozo por mes. La timeline mensual es lo que permite generar lag features.
VERDADERO"Después del preprocesamiento, las categóricas son enteros." → Sí, LabelEncoder produce int32. No se usa one-hot (los RF lo manejan bien).
📝 Preguntas tipo examen
¿Por qué el RFC dice que "el foco está en ML Engineering, no en los modelos"?▾
Porque la materia es "IA en Producción". La habilidad evaluada no es construir el mejor modelo predictivo sino diseñar el pipeline completo: ingesta, features, training reproducible, tracking, deployment, monitoring. Un modelo perfecto sin trazabilidad ni capacidad de update es inútil en producción.
¿Por qué usar LabelEncoder y no OneHotEncoder para las categóricas?▾
Los Random Forest manejan naturalmente variables categóricas como enteros porque hacen splits basados en valor, no asumen orden ni magnitud. OneHot multiplicaría la dimensionalidad (provincia podría tener 20+ valores) sin aportar al modelo. Para modelos lineales o redes sí necesitaríamos OneHot.
¿Qué pasa si el dataset agrega un pozo nuevo después de que el modelo está en producción?▾
El nuevo idpozo entra al feature store en la próxima corrida de build_feature_store. La API consulta Feast por id en tiempo real, así que automáticamente lo serviría. Pero atención: si las categóricas (provincia, cuenca) tienen valores nuevos, el LabelEncoder fitteado anteriormente puede romperse — habría que fitearlo siempre con el dataset completo (que es lo que pasa hoy).

El stack tecnológico
5 componentes principales. Cada uno con su rol específico y trade-offs.
🟠 Orquestación
Airflow 3.0.6 CeleryExecutor
⌄
Orquesta los 4 DAGs del proyecto. Scheduler + workers + api-server corriendo en containers separados.
💡 Por qué Airflow vs cron
Cron solo dispara comandos en horarios fijos. Airflow agrega: dependencias entre tasks (un DAG es un grafo dirigido), retries automáticos, UI para monitorear, backfills (correr el DAG para fechas pasadas), params dinámicos al disparar, logs centralizados por task, y schedules complejos. Cron no tiene nada de eso.
🧩 Componentes (containers separados)
  • airflow-apiserver (puerto 8181) → la UI web y API REST.
  • airflow-scheduler → decide qué tasks correr y cuándo.
  • airflow-worker (Celery) → ejecuta los tasks. Pueden ser N.
  • airflow-dag-processor → parsea los archivos .py de los DAGs.
  • airflow-triggerer → maneja sensors/deferrable tasks.
  • postgres → base de datos de metadata (estado de runs, tasks, etc.).
  • redis → broker de Celery (cola de tasks).
📜 Qué es un DAG
Directed Acyclic Graph. Un archivo Python donde declarás tasks y sus dependencias. En este proyecto: 4 DAGs en dags/ usando el decorador @dag y @task de Airflow 3.
VERDADERO"Si un task del DAG falla, el siguiente no se ejecuta." → Sí (por defecto). Las dependencias se respetan: task_b solo corre si task_a termina en estado success.
FALSO"Los DAGs corren automáticamente al subirlos." → No. Hay que despausarlos en la UI primero. Por seguridad: AIRFLOW__CORE__DAGS_ARE_PAUSED_AT_CREATION: 'true'.
🟣 Tracking + Registry
MLflow 3.10.1
⌄
Tracking de experimentos + Model Registry. Backend SQLite, artifacts en filesystem montado.
💡 Tracking vs Registry
Son dos cosas distintas:
  • Tracking: "qué corrí, con qué params, qué métricas dio". Cada training es un run dentro de un experiment.
  • Registry: "qué modelo está en producción". Versiones nombradas con aliases (production, staging).
🏷️ Aliases vs versiones numéricas
Cada vez que registrás un modelo, MLflow le da una versión numérica (1, 2, 3...). Pero la API no quiere conocer números — quiere "el modelo que está en producción ahora". Solución: el alias production apunta siempre a la versión actual. Si promovés v8 → alias se mueve a v8. La API hace models:/gas_model@production y siempre obtiene la última.
📦 Configuración del compose
--backend-store-uri sqlite:///mlflow/mlflow.db → SQLite local para metadata.
--default-artifact-root /mlflow/mlruns → directorio compartido para modelos y plots.
Volumen ./mlruns montado en API y workers → todos comparten los artifacts.
FALSO"Si borrás el alias, se borra el modelo." → No. El alias es solo un puntero. Borrar el alias deja la versión intacta, lista para reasignar.
🟢 Feature Store
Feast 0.54 Redis + Parquet
⌄
Offline store en Parquet (para training), online store en Redis (para inferencia low-latency).
💡 Por qué un feature store
Sin feature store, las features se recalculan en cada lugar (training, inferencia, validación) — y eventualmente divergen (un lag se calcula distinto en training que en inferencia → train/serve skew). El feature store es una sola fuente de verdad para features.
🔄 Online vs Offline
  • Offline (Parquet): dataset histórico completo, para training. Se accede con store.get_historical_features() o leyendo directo el parquet.
  • Online (Redis): última fila por entidad (pozo), para inferencia en tiempo real. Latencia < 5ms. Se accede con store.get_online_features(entity_rows=[{"idpozo": X}]).
🎯 Por qué Redis
  • Baja latencia: in-memory, ideal para serving real-time.
  • Soportado nativamente por Feast.
  • Fácil de levantar con Docker.
  • Escalable (Redis Cluster) si el sistema crece.
  • Alternativas: DynamoDB (AWS, serverless), BigTable (GCP), Cassandra. Trade-off: más complejas de operar localmente.
FALSO"Cada pozo guarda toda su historia en Redis." → No. Solo se guarda la última fila por pozo en el online store. La historia completa vive en el parquet offline.
🔴 Serving
FastAPI + Ray Serve 2.20
⌄
FastAPI define los endpoints. Ray Serve los despliega con múltiples réplicas para distribuir carga.
💡 Por qué Ray Serve
FastAPI solo es un servidor web async. Si querés escalar horizontalmente, necesitás replicar la app y poner un load balancer adelante. Ray Serve hace eso integrado: declarás num_replicas=3 y Ray levanta 3 procesos worker con el modelo cargado en cada uno, balancea requests entre ellos, y los maneja como un cluster.
🧩 La declaración
@serve.deployment(num_replicas=3)
@serve.ingress(app)
class API: ...
El decorator ingress conecta la FastAPI app al deployment. Cada request entra al pool, Ray la rutea a una réplica libre.
⚡ Cumple el requerimiento RFC
"El sistema DEBE implementar una arquitectura escalable para responder la inferencia (ej: Ray)." → Cumplido con num_replicas=3.
VERDADERO"Cada réplica tiene su propio modelo cargado en memoria." → Sí. Si el modelo pesa 200MB y tenés 3 réplicas, usás 600MB de RAM. Trade-off: más memoria por mejor throughput.

📝 Preguntas tipo examen — stack
¿Por qué Airflow y no un cron + scripts?▾
Por dependencias entre tasks (un DAG es un grafo), retries automáticos, UI para monitorear, params dinámicos en cada run, logs centralizados, backfills para fechas pasadas. Cron no tiene nada de eso. Para un sistema productivo con dependencias entre pasos, Airflow es estándar.
¿Qué es un alias en MLflow y por qué usarlo en lugar de la versión numérica?▾
Un alias es un nombre simbólico que apunta a una versión específica de un modelo registrado. La API hace models:/gas_model@production en lugar de models:/gas_model/8. Si después se promueve la v9, basta con mover el alias — la API no necesita cambiar código ni redeployarse. Es el equivalente al "tag" en Git: separa nombre simbólico de versión concreta.
¿Por qué Redis como online store? ¿Qué alternativas hay?▾
Redis por: baja latencia (in-memory, <5ms), soportado nativamente por Feast, fácil de levantar con Docker, escalable. Alternativas: DynamoDB (AWS, serverless, paga por uso), BigTable (GCP, optimizado para alta escritura), Cassandra (on-prem distribuido). En cloud serverless se prefiere DynamoDB; en proyectos académicos o startups, Redis es estándar por simplicidad.
¿Qué problema resuelve un feature store que un parquet común no resuelve?▾
Tres cosas: (1) Train/serve skew: si las features se calculan en código diferente para training e inferencia, divergen. El feature store es la fuente única. (2) Versionado de features: cambiar la definición de una feature impacta a todos los modelos que la usan. (3) Latencia: el offline store tiene todo (parquet, lento de filtrar) pero el online da O(1) lookup por entidad.
¿Por qué docker-compose y no Kubernetes?▾
Por scope académico. Compose levanta todo en local con un comando, es trivial de leer, y permite developing rápido. Kubernetes agrega complejidad (manifests, services, ingress, persistent volumes, deployments separados) que solo se justifica en producción real. Para una entrega académica, compose es la elección correcta.

🗺️ El flujo end-to-end
📖 Cómo leer este diagrama Cuatro DAGs en horizontal forman la línea principal de producción. El cuarto (drift, abajo en rosa) es transversal: consume artefactos de los otros tres.

1. build_feature_store (naranja): descarga el CSV, genera features con lags y rolling averages, popula Redis. Output: parquet + snapshot timestamp.

2. ml_training_pipeline (violeta): lee el YAML de experimentos, entrena un Random Forest por target (gas, petróleo), loguea métricas en MLflow.

3. automatic_model_selection (verde): elige el mejor run según métrica (default: MSE ASC), lo registra como versión y le pone alias production.

API (rojo): al arrancar y cada 60s detecta cambios de alias y recarga los modelos sin restart. Sirve /forecast, /wells.

4. drift_and_decay_report (rosa, transversal): compara snapshots del feature store en dos momentos + re-evalúa el modelo productivo. Output: reporte HTML en MLflow.

📋 Sobre el factor de decay 0.97n
El modelo predice solo el mes inmediatamente siguiente al último observado. Para meses más lejanos se aplica un factor de decaimiento empírico: base × 0.97^n donde n es la distancia en meses. Es una aproximación pragmática — no es ML real, es heurística de negocio basada en el supuesto de que la producción decae mensualmente en pozos no convencionales.
🔄 Hot reload: actualizar modelo sin restart
Cuando se promueve un modelo nuevo a production en MLflow, la API debe detectarlo y recargar — pero sin reiniciar el container ni desconectar usuarios.
⚙️ Mecanismo
  • self.current_versions guarda el run_id del modelo cargado actualmente.
  • maybe_reload() se llama en cada /forecast pero hace early return si pasaron menos de 60s desde la última verificación (cooldown).
  • Si pasó el cooldown: model_version_changed() consulta MLflow por el run_id actual de @production. Si difiere, recarga.
  • Adicionalmente hay un endpoint POST /reload-model para forzar manual.

📝 Preguntas tipo examen — API + Ray
¿Qué pasa exactamente con num_replicas=1 vs num_replicas=3?▾
Con 1 réplica, todas las requests entran a un solo proceso. Si una request tarda 200ms en hacer la inferencia, otra request bloquea hasta que termine (FastAPI async ayuda con I/O pero no con CPU). Con 3 réplicas, hay 3 procesos worker en paralelo — Ray rutea por round-robin. Throughput ~3x. Costo: 3x la RAM del modelo cargado.
¿Por qué el endpoint /wells lee directamente el parquet y no el online store?▾
Porque necesita listar TODOS los pozos activos en un rango temporal, no consultar uno solo. El online store de Feast está diseñado para lookups O(1) por entidad, no para queries por rango. Para queries analíticas sobre todos los pozos, el parquet (offline) es lo apropiado.
¿Qué es el factor de decay 0.97n y qué supuesto hace?▾
Es una heurística que multiplica la predicción del modelo por 0.97^n donde n es meses adelante. Supone que la producción decae 3% por mes — típico de pozos no convencionales (shale) donde la curva de declinación es agresiva en los primeros años. No es ML, es business knowledge. Una alternativa sería entrenar un modelo multi-step con los targets forward (prod_*_f1, prod_*_f2) ya generados en el feature store.
¿Cómo se entera la API que el modelo cambió?▾
Polling con cooldown. En cada request a /forecast, maybe_reload() chequea (si pasaron 60+ segundos) si el run_id del alias production en MLflow cambió. Si sí, recarga. También hay POST /reload-model para forzar. Trade-off: hay una ventana de hasta 60s donde la API sirve el modelo viejo. Para producción real se usaría webhook o event-driven.
¿Por qué las features del modelo se guardan como param string en MLflow y no en código?▾
Para desacoplar la API del experimento. La API no sabe qué features usa cada modelo — las lee del run con ast.literal_eval(run.data.params["features"]). Si mañana entrenás un modelo con feature set distinto y lo promovés, la API automáticamente usa las nuevas features. Cero cambio de código.

Acto 5
Monitoring · drift detection + model decay
El requerimiento crítico de la entrega final. Dos métricas para drift + una métrica de decay = reporte completo.
📊 Data Drift ⌄
Cambia la distribución de las features de input. El modelo recibe datos fuera de su entrenamiento.
💡 Qué es
La distribución marginal de una feature cambió entre el training y el momento actual. Ejemplo: en training, tef tenía media 24h. Después se empezaron a registrar pozos con tef de 6h. El modelo no vio eso — la predicción se vuelve incierta.
🛢️ Aplicado a oil & gas
  • Una cuenca nueva entra al dataset → distribución de provincia/cuenca cambia.
  • Empresas operadoras cambian políticas de medición → tef se mueve.
  • Pozos nuevos vs pozos maduros → distribución de profundidad se sesga.
  • Macro económico → empresas cierran pozos viejos → distribución de targets se mueve.
🎭 Concept Drift ⌄
La relación entre features y target cambia. Las mismas X ya no producen el mismo Y.
💡 Diferencia con data drift
Data drift: P(X) cambió. Concept drift: P(Y|X) cambió. Son distintos.
🛢️ Ejemplo
Mismo pozo, misma profundidad, mismas categóricas. Pero después de una optimización operativa (estimulación química), la producción real ahora es 1.5x lo que el modelo predice. El input no cambió. La relación sí.
FALSO"Concept drift se detecta con KS test." → No, el KS test detecta data drift (cambios en distribución de features). Concept drift se detecta comparando predicción vs realidad — eso es lo que mide el model decay.
📉 Model Decay ⌄
El modelo empeora medido por métricas de performance (MSE sube, R² baja). Es el síntoma, no la causa.
💡 Qué es
Re-evaluar el modelo productivo sobre datos nuevos con ground truth. Si MSE sube significativamente vs training, hay decay. Puede ser causado por data drift, concept drift, o cambios sin explicar.
🔧 Cómo se calcula en este proyecto
  • Cargar gas_model@production y pet_model@production.
  • Tomar el snapshot actual del feature store con ground truth (filas con target no-null).
  • Predecir y calcular MSE + R².
  • Comparar contra run.data.metrics["mse"] y r2 del entrenamiento original.
  • Delta: mse_current − mse_train, r2_current − r2_train. Si |r2_delta| > 0.1 → veredicto "REVISAR MODELO" en el reporte HTML.
📐 Las dos métricas de drift implementadas
El RFC pide "al menos 2 métricas". Implementamos KS Drift (univariada) + ClassifierDrift (multivariada). Son complementarias: una detecta cambios en cada feature por separado, la otra detecta cambios de correlación.
🟢 KS Drift — test de Kolmogorov-Smirnov por feature
⌄
Compara las CDFs de cada feature entre referencia y actual. p-value < 0.05 → drift en esa feature.
💡 Intuición
Para cada feature, comparamos la CDF empírica en el snapshot referencia vs el actual. La distancia máxima vertical entre las dos curvas es el estadístico D. Si las distribuciones son iguales, D ≈ 0. Si son diferentes, D crece. El p-value mide qué tan probable es ver ese D bajo la hipótesis nula (mismas distribuciones).

⚠️ Limitación clave
KS es univariado: testea cada feature aisladamente. Si prod_gas_t1 y prod_pet_t1 tienen las mismas marginales pero se descorrelacionaron, KS no lo detecta. Para eso necesitamos ClassifierDrift.
🟣 ClassifierDrift — entrenar un detector binario
⌄
Entrena un RandomForest a distinguir "referencia" (label 0) vs "actual" (label 1). Si AUC ≫ 0.5, las distribuciones son distinguibles → drift.
💡 Intuición
Reformulamos drift detection como clasificación binaria. Si un modelo puede distinguir si un sample viene de referencia o actual con AUC alta, las distribuciones son distinguibles → hay drift. Bonus: las feature importances del clasificador te dicen exactamente qué features cambiaron más.

📝 Preguntas tipo examen — monitoring
¿Cuál es la diferencia entre data drift, concept drift y model decay?▾
Data drift: P(X) cambia — la distribución de las features de input se mueve. Concept drift: P(Y|X) cambia — las mismas features ahora producen distinto target. Model decay: la performance (MSE, R²) empeora. Decay es el síntoma; drift (data o concept) son causas posibles. Otra causa: sesgo en la selección de datos.
¿Por qué necesitás KS + ClassifierDrift? ¿No alcanza con uno?▾
Son complementarios. KS testea cada feature aisladamente con su CDF — rápido y direccional (te dice cuál feature). Pero no ve correlaciones: si las marginales son iguales pero la relación entre features cambió, KS dice "todo OK". ClassifierDrift sí lo ve, pero es un test agregado (no te dice cuál feature). Juntos: KS por feature + Classifier para el conjunto. Si tuviera que elegir una, ClassifierDrift es más robusta pero menos interpretable.
¿Por qué hay que filtrar dropna(subset=[target]) antes de calcular decay?▾
El feature store agrega una "inference row" por pozo (la fila T+1 sintética que sirve para que la API tenga lags listos). Esa fila tiene target = None porque T+1 todavía no se observó. Si no se filtra, el cálculo de MSE/R² incluiría NaNs y reventaría. El filtrado deja solo las filas con ground truth real.
¿Por qué publicar a MLflow en lugar de generar archivos sueltos?▾
Centralización. MLflow ya es el sistema donde viven los modelos y sus métricas de training. Tener las métricas de monitoring ahí permite (1) compararlas históricamente, (2) verlas en la misma UI, (3) tener un único punto donde mirar el "estado" del sistema. Si después se quisiera agregar alerting o un dashboard, MLflow API es la fuente. Y los artifacts HTML se descargan desde la UI sin cambios.
¿Qué sería un "concept drift" en este proyecto y cómo lo detectarías?▾
Concept drift sería: misma profundidad, mismas categóricas, mismos lags — pero la producción real ya no es la que el modelo predice (por ejemplo, mejoras tecnológicas en la operación). No se detecta con KS (que solo ve P(X)). Se detecta con model decay: si MSE sube significativamente pero el data drift es bajo, es señal de concept drift. Es exactamente lo que hace nuestro reporte: combinamos las tres mediciones para distinguir.
¿Qué snapshot usar como referencia?▾
Idealmente, el snapshot con el que se entrenó el modelo en producción. El DAG acepta el param reference_snapshot: con Earliest usa el más viejo disponible; con una fecha AAAAMMDD usa el snapshot específico. Para alinear exactamente con training, lo ideal es leer el param fecha_de_data del run productivo en MLflow. Es una mejora pendiente.
